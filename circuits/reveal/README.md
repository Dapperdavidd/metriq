# circuits/reveal — the Noir reveal (Ring 3, the cut line)

Prove the win, hide the route. At settle, the winner posts one Noir proof: it reached
the quality target under the cap via a route matching a commitment posted before the
run, WITHOUT revealing the route. `RevealVerified` is the only event the frontend ever
needs from Ring 3, so cutting it removes exactly one component (`ProofTick`).

This is **not a scaffold**: the proof is real, generated by Barretenberg, and verified
on chain by the bb-generated UltraHonk Solidity verifier. See
`contracts/test/Reveal.t.sol` (3 passing tests: the real proof verifies on chain, the
reveal emits the green tick, a tampered public input is rejected).

## The circuit

`src/main.nr`. Private witness: the route (which tier per subtask), its prices, its
scores, and a salt. Public inputs: the commitment, the cap, the spend (already public
from the `Charged` events), and the quality target. It asserts:

1. the private plan hashes to the pre-run commitment (Pedersen),
2. the prices sum to the public spend and stay within the cap,
3. the hidden route actually met the target.

Hardening beyond the source draft: the commitment binds route **and prices and scores**
(not just route + salt), so a prover cannot keep the committed route but claim inflated
scores.

## Toolchain

- **Noir** `nargo` 1.0.0-beta.22 (install: `noirup`)
- **Barretenberg** `bb` 5.0.0-nightly.20260522 (install: `bbup`, which matches your nargo)

## Reproduce the proof and verifier

```bash
cd circuits/reveal
nargo test                 # the circuit's own unit test
nargo compile              # -> target/reveal.json (ACIR)
nargo execute witness      # -> target/witness.gz (uses Prover.toml)

bb write_vk  --scheme ultra_honk -b ./target/reveal.json -o ./target --oracle_hash keccak
bb prove     --scheme ultra_honk -b ./target/reveal.json -w ./target/witness.gz -o ./target --oracle_hash keccak
bb verify    --scheme ultra_honk -k ./target/vk -p ./target/proof -i ./target/public_inputs --oracle_hash keccak

# the Solidity verifier (already committed at contracts/src/HonkVerifier.sol):
bb write_solidity_verifier --scheme ultra_honk -k ./target/vk -o ./target/Verifier.sol
```

The `proof`, `public_inputs`, and `vk` are committed under
`contracts/test/fixtures/reveal/` so the on-chain test runs on a fresh clone without the
toolchain. `Prover.toml` holds the winning route's witness; the `commitment` in it is the
Pedersen hash of that witness (printed once from the circuit's test and pinned).

## Deploy note

The generated `HonkVerifier.sol` uses `require(bool, CustomError())`, which needs
**solc 0.8.26+** — this is why `contracts/foundry.toml` pins `solc_version = "0.8.27"`.
Our own contracts stay `pragma ^0.8.24`; only the compiler moved, for the third-party
verifier. The UltraHonk verifier is large (~100 KB) and exceeds the EIP-170 code-size
limit, so a mainnet deploy would need the verifier split or an L2 that relaxes the limit;
on X Layer testnet and anvil it deploys as-is.

## Cut-line discipline (decided at the July 12 checkpoint)

If the core demo is not stable at the checkpoint, Ring 3 does not ship: delete
`apps/web/src/components/ProofTick.tsx` and drop the `revealVerifier` address from the
indexer config. Nothing else changes.
